11/** \file dc/pvr.h
12 \brief Low-level PVR (3D hardware) interface.
13 \ingroup pvr
14
15 This file provides support for using the PVR 3D hardware in the Dreamcast.
16 Note that this does not handle any sort of perspective transformations or

18 hardware support.

21 management to actual primitive rendering.
22

516/** \defgroup pvr_uv_flip U/V Flip Mode
517 \brief Enable or disable U/V flipping on the PVR
518 \ingroup pvr_ctx_texture
519
520 These flags determine what happens when U/V coordinate values exceed 1.0.
521 In any of the flipped cases, the specified coordinate value will flip around
522 after 1.0, essentially mirroring the image. So, if you displayed an image

977
978 \param a Alpha value
979 \param r Red value
980 \param g Green value
981 \param b Blue value
982 \return The packed color value
983*/

985 ( ((uint8_t)( (a) * 255 ) ) << 24 ) | \
986 ( ((uint8_t)( (r) * 255 ) ) << 16 ) | \
987 ( ((uint8_t)( (g) * 255 ) ) << 8 ) | \
988 ( ((uint8_t)( (b) * 255 ) ) << 0 ) )

990/** \brief Pack two floating point coordinates into one 32-bit value,
991 truncating them to 16-bits each.
992
993 \param u First coordinate to pack
994 \param v Second coordinate to pack
995 \return The packed coordinates
996*/
997static inline uint32_t PVR_PACK_16BIT_UV(float u, float v) {
998 union {
999 float f;
1000 uint32_t i;
1001 } u2, v2;
1002
1003 u2.f = u;
1004 v2.f = v;
1005
1006 return (u2.i & 0xFFFF0000) | (v2.i >> 16);
1007}

1747/* Scene rendering ***************************************************/
1748/** \defgroup pvr_scene_mgmt Scene Submission
1749 \brief PowerVR API for submitting scene geometry
1750 \ingroup pvr
1751
1752 This API is used to submit triangle strips to the PVR via the TA
1753 interface in the chip.
1754
1755 An important side note about the PVR is that all primitive types
1756 must be submitted grouped together. If you have 10 polygons for each
1757 list type, then the PVR must receive them via the TA by list type,
1758 with a list delimiter in between.
1759
1760 So there are two modes you can use here. The first mode allows you to
1761 submit data directly to the TA. Your data will be forwarded to the
1762 chip for processing as it is fed to the PVR module. If your data
1763 is easily sorted into the primitive types, then this is the fastest
1764 mode for submitting data.
1765
1766 The second mode allows you to submit data via main-RAM vertex buffers,
1767 which will be queued until the proper primitive type is active. In this
1768 case, each piece of data is copied into the vertex buffer while the
1769 wrong list is activated, and when the proper list becomes activated,
1770 the data is all sent at once. Ideally this would be via DMA, right
1771 now it is by store queues. This has the advantage of allowing you to
1772 send data in any order and have the PVR functions resolve how it should
1773 get sent to the hardware, but it is slower.
1774
1775 The nice thing is that any combination of these modes can be used. You
1776 can assign a vertex buffer for any list, and it will be used to hold the
1777 incoming vertex data until the proper list has come up. Or if the proper
1778 list is already up, the data will be submitted directly. So if most of
1779 your polygons are opaque, and you only have a couple of translucents,
1780 you can set a small buffer to gather translucent data and then it will
1781 get sent when you do a pvr_end_scene().
1782
1783 Thanks to Mikael Kalms for the idea for this API.
1784
1785 \note
1786 Another somewhat subtle point that bears mentioning is that in the normal
1787 case (interrupts enabled) an interrupt handler will automatically take
1788 care of starting a frame rendering (after scene_finish()) and also
1789 flipping pages when appropriate.
1790*/
